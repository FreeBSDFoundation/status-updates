## **Reporting Period:** [2026-05-04] - [2026-05-10]

### Project: Full CPUID Control for bhyve
#### **Overall Status:** On Track
Began working on more thorough CPUID feature clearing and implementing CPU
types/families. Realized that I should probably go through feature bits as
I encounter them in CPU types/families and determine into which of these
categories it belongs:

- immutable, must not change from how vmm emulated them (e.g. vendor
  identification, HVM features)
- may only be cleared by the user (e.g. instruction set extensions, HW
  vulnerability mitigations)
- may be set or cleared by the user at will

It also occured to me that there's a reasonable number of well-known features
for ISA extensions, and that it would be nice if bhyve understood their names
rather than insisting on them being configured by bit manipulation. There are
of course, also a number of CPUID leaves that expose more complex information
such as CPU and cache topology that probably don't this.

Hypervisor signature override may require changing through a separate vmm ioctl
interface and require an additional CPUID emulation step in vmm to properly
indicate supported features of other hypervisors.
This may mean that overriding CPUID leaves in the 0x40000000 range should be
forbidden completely.
