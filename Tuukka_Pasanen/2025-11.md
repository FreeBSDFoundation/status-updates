with Alpha-Omega database.yml data adding main challenge was determining the upstream fork point (over 50% of the time this can be traced from commit messages). However, this isn't the core problem. Here's a breakdown:

Imagine FreeBSD forks library `libfoo` version 1.0.0 (with an acceptable BSD-2-Clause license) into `src/contrib/libfoo`. A vulnerability is then discovered in libfoo 1.0.0 and documented as CVE-XXXXX. FreeBSD creates a patch, commits it to its own Git repository, and submits the fix upstream. However, if the upstream maintainer is slow to review and integrate the change, `libfoo` in FreeBSD diverges from the original version (although it's still a soft fork, and FreeBSD integrates new versions when they are released).

When FreeBSD releases version YY.X, which includes libfoo, there's an issue with communication. How do you communicate that CVE-XXXXX is fixed in FreeBSD but not upstream? The OSV vulnerability database can help, but how do you determine this in the SBOM (Software Bill of Materials)?

Should these forked projects be versioned as FreeBSD YY.X or retain their upstream version number with a patch number?
