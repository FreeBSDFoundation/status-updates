### `mac_do(4)`/`mdo(1)`
Still some work to put the GSoC 2025's "executable paths" task in shape for
commit.

- A thorough review of the consistency currently provided for concurrent changes
  to `mac_do(4)`'s jail configurations shows it is too weak, making the use of
  inheritance dangerous from a security standpoint in some situations (this
  problem has been there from the start, it is not new with the executable paths
  feature).
- Try to fix it in the framework of fine-grained locking (each configuration
  structure is protected by its jail's lock), but finally give up on it as,
  although this is feasible, it would required even more changes than I had
  envisioned and done so far, and the complexity is not worth it given that
  configuration changes are expected to be rare.
- Switch to using a single read-mostly lock to protect all configuration
  objects. Impacts in some of the fixes to the "executable paths"
  feature. Almost finished, to be tested.
