## **Reporting Period:** 2026-04-08 - 2026-04-14

### Power / Suspend-to-idle (S0ix) \[Laptop Project\]
- [D55118](https://reviews.freebsd.org/D55118) ("acpi: Parse _S3D s2idle instead
  of _S255D"): Approve after latest changes.
- Read the ACPI specification about `_LPI` objects (seen on some AMD machines)
  and the separate Intel specification about the ACPI `LPIT` table (seen on some
  Intel machines).  These contain information about Low-Power Idle states.
- A priori we do not really need to implement support for `_LPI` objects on AMD
  platforms to support S0ix at this point (at least for our target platform), as
  it seems that Low-Power Idle states and C states in ACPI are just in the end
  mapped to the same processor C states (and ACPI also allows
  backwards-compatibility for OSes that do not support `_LPI` objects and other
  ones related to Low-Power Idle) by tolerating idle states in `_CST` objects).
  However, this needs confirmation, e.g., by examining the ACPI tables and
  namespace on more machines, and perhaps also by having FreeBSD indicate
  support via `_OSC`.  Besides an "entry point" and residency counter
  information, `_LPI` objects have an "Arch. Context Lost Flags", which
  fortunately must always be 0 on `amd64` (contrary to ARM), and a way to convey
  information on how to request "composite states" (where the system/package
  levels above cores could be requested separately, if not "autopromoted").
- The `LPIT` tables we have seen in practice do not seem to be filled completely
  correctly.  In particular, Entry Triggers do not match any `_CST` object's
  register, and the Residency Counter Frequency may not be filled correctly
  either.  The safer solution here to get residency counters is to implement
  a driver.
- Start reviewing [D54881](https://reviews.freebsd.org/D54881) ("intel/intelpmc:
  Add Intel PMC Core driver"), whose main purpose is to provide information on
  entering Low-Power Idle states, in particular give access to the lowest
  processor's C state residency counter.  The drawback of this approach is that
  we need specific code/constants for each generation of Intel PCH, but it seems
  we don't have a solid alternative, given the `LPIT` tables we have seen in
  practice (as explained above).  Read part of the corresponding Linux code to
  have an idea of where this would head.
- Work in progress on `acpi_spmc(4)`, aimed at giving clearer debugging
  information, such as human-readable presentation of which DSMs are present and
  which methods are reported on them.  Some minor fixes.
- Testing on my Framework Intel laptop (VT use, no Xorg nor Wayland):
  - Idle consumption, no DRM driver: ~700 to 800mA.
  - S0ix, no DRM driver: Screen does not turn off, the power button blinks.
    A 30-min S0ix test does not show any significant battery drain difference
    with idle S0.
  - Idle consumption, with DRM driver: ~300 to 400mA.
  - On S0ix entry, with DRM driver (6.9 (equivalent to
    `graphics/drm-latest-kmod`) or `main`) screen turns black (but is still
    powered) and the laptop seems to hang completely.

### Misc
- Create [D56227](https://reviews.freebsd.org/D56227) ("netstat(1): On '-w',
  print statistics also on banner printing"): Initially, I thought that, on each
  new banner printing, the statistics line was omitted, causing an uneven time
  series to be reported. Fix change of behavior concerning the first statistics
  line reported by `glebius@` (it would be output immediately, and be full of
  0 most of the time, as the two samples were taken without waiting for an
  interval). After more investigation, a statistics line was indeed printed on
  banner printing, but before it, which was very confusing. The change here
  ensures that a new banner is printed only if a new statistics line is going to
  be printed just after it.
