#### **Features Delivered**
- See [kdb.md](kdb.md)
#### **Challenges**
- Testing on non-x86 and non-arm64 was difficult, but now I have a laptop with QEMU and ppc64le server (not set up yet).
#### **Blockers**
- I need jhb@ to comment on three pull requests, otherwise I cannot create my next PRs.
#### **Next Steps**
- Add missing error checks in `ProcessFreeBSDKernelCore`.
- Create a PR for trapframe unwinding after jhb@ finishes reviewing existing PRs.
- Migrate kgdb python scripts to LLDB (see below to see why and how).
- Finish minidump-to-elf (will be renamed to md2elf).
#### **Open Questions and Discussion**
- LLDB's stack unwinding isn't as flexible as GDB's. Thus there is no support for i386 and armv7 due to LLDB's limitation.
- At first, I was going to migrate scripts under `sys/tools/gdb` in Lua or Python. However this causes cross-platform issue that
  not all package managers will ship our custom scripts with their packages. So I think it would be better to implement those scripts
  in native C++ and upstream it to `llvm/llvm-project`.
- LLDB doesn't support functions yet (e.g. `print $PCPU("pc_curthread")`) and this isn't a problem that is specific to a
  programming language (C++, Python, Lua). Until LLDB implements support for custom function, we can only have a workaround by
  `process plugin pcpu pc_curthread` then play with the value to do other stuff (e.g. check values in `pc_curthread`).
